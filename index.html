01 What is this?

The Nabaztag was one of the first consumer "smart" objects — a plastic rabbit with motorized ears, released in 2005. Its cloud servers died long ago, so this project brings one back to life using the open-source pynab firmware running on a Raspberry Pi Zero W hidden inside it.

The result is a small appliance that does three things:

🔊 Plays audio

A tagtagtag WM8960 sound HAT turns the rabbit into a real speaker with left/right channels.

🎵 Spotify Connect

It shows up in the Spotify app as a speaker named Nabaztag — pick it and music streams to it.

👂 Wiggles its ears

A hook fires on every song change and nudges the motorized ears for a little dance.

💾 Is fully documented

This repo is the source-of-truth backup for every script and setup note.

02 How it all fits together

Music comes in over Wi-Fi from Spotify. spotifyd decodes it and plays it through the sound HAT. On every track change it also runs a hook script, which asks the nabd daemon to move the ears.

Firmwarepynab — the nabd daemon plus a web config UI
AccessSSH over the LAN with key-based login

The Pi Zero W is deliberately modest — that constraint drives most of the interesting decisions below (old glibc, no MP3 decoder, tiny CPU), so everything leans on lightweight, stdlib-only tools.

04 The brain: pynab & nabd

pynab is the community firmware that replaces the dead 2005 cloud service. Its core is nabd, a daemon that owns the hardware — the ears, the LEDs, the RFID reader — and exposes them over a simple JSON-over-TCP protocol on 127.0.0.1:10543.

You control the rabbit by opening a socket and sending one JSON object per line. To move the ears you send an ears command with absolute positions (the ears rotate through ~17 discrete steps, where 0 is rest):

# one line of JSON = one command
{"type":"ears", "left":3, "right":3}

05 The ear wiggle

Two small scripts create the signature effect.

wiggle_ears.py

Opens the socket to nabd, swallows its greeting, then sends a short sequence of ear positions — out, back, out, back — pausing ~0.45s between each so the motors have time to travel. It's written to fail silently: if nabd is busy or unreachable it just exits, because it gets spawned constantly.

on_song_change.sh

The spotifyd hook. spotifyd sets a PLAYER_EVENT variable on every player event; this script only wiggles on change/start/play — never on stop or pause — and launches the wiggle in the background so it never blocks the audio thread.

# wired up once in /etc/spotifyd.conf
on_song_change_hook = "/home/pi/on_song_change.sh"

06 Spotify Connect on a tiny, old Pi

The speaker side runs spotifyd, a lightweight open-source Spotify Connect daemon. Getting it running on a Pi Zero W was the fiddly part:

  • Old CPU, old libc. The Pi Zero W is ARMv6 with glibc 2.28, so it needs the specific spotifyd v0.3.3 armv6-slim build — newer spotifyd wants glibc ≥ 2.29 and raspotify is out entirely.
  • No stored password. Spotify killed password auth for old librespot, so login is via zeroconf/discovery — you activate it once by picking "Nabaztag" in a real Spotify app on the same Wi-Fi, and it's remembered.
  • Runs as a service. spotifyd.service is enabled and auto-starts; audio is routed to the HAT via plughw:CARD=tagtagtagsound.

07 Getting sound out of it

Buster is end-of-life and has no working MP3 decoder (mpg123 is uninstallable), and aplay only understands WAV. Two scripts work around that.

test_audio.py · runs on the Pi

A stdlib-only tone generator — no numpy, no pyaudio. It synthesizes a sine wave (with a 20 ms fade to avoid clicks), writes a WAV, and plays it through ALSA. Great for a first "does the speaker work?" check.

python3 ~/test_audio.py --sweep
play_mp3.sh · runs on the Mac

Sidesteps the missing decoder entirely: it converts the MP3 to WAV on the Mac with the built-in afconvert, copies it over with scp, and plays it remotely over SSH.

./play_mp3.sh ~/Downloads/song.mp3
